# R05-CAN-001 — Address geocoding query strings cross retention boundaries

- Candidate ID: R05-CAN-001
- Title: Address geocoding query strings cross retention boundaries
- Instance key: address-query-disclosure:frontend/src/components/FacilityMap.tsx:271
- Ledger row ID: central-validation-round-06:R05-CAN-001

## Attack Path Steps

1. Source: The sensitive value is a victim-entered address; the relevant recipients and possible log readers are the application infrastructure and Naver provider.
2. Control gap: GeocodeController trims, rejects blank values, and caps length at 200; GeocodeClient deliberately avoids logging WebClient exceptions because they may include the address, but the UI gives no provider-sharing or retention disclosure.
3. Sink: The address appears in both application and provider request URLs before results are returned.
4. Impact: A precise home/work/care address can be retained in access logs or provider telemetry and linked to a health-service session despite the user only intending to choose a map centre.
