## Attack Path Steps

1. Attacker or external source controls the validated input: An external model controls emergency action label, phone, href, and even omission/null combinations within the provider schema.
2. The stated preconditions hold: The model emits emergency urgency plus a structurally valid but incorrect or unusable SHOW_EMERGENCY_CALL action.
3. The closest control is reached but does not close the path: The validator checks only that a SHOW_EMERGENCY_CALL subtype exists, not its canonical payload.
4. The input reaches the validated sink: The backend accepts an arbitrary action that the active ChatScreen does not render.
5. The resulting consequence is: Emergency output can pass validation without providing the authoritative 119 dialing affordance promised by the safety contract.
