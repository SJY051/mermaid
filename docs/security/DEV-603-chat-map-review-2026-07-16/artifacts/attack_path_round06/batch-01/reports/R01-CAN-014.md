## Attack Path Steps

1. Attacker or external source controls the validated input: A contributor must add browser code that actively reads a key-bearing VITE_ variable, and an operator must populate that same variable with a real server credential.
2. The stated preconditions hold: Production frontend code actively references the evading VITE_ name, the build environment supplies a real secret under that exact name, and the generated bundle is published.
3. The closest control is reached but does not close the path: The Vite build guard inspects every loaded VITE_ name but denies only names matching SECRET, PASSWORD, PASSWD, PRIVATE_KEY, TOKEN, or CREDENTIAL.
4. The input reaches the validated sink: Only an actively referenced expression such as import.meta.env.VITE_LLM_API_KEY is statically replaced into the shipped bundle; merely defining an unreferenced environment variable is not a demonstrated bundle sink.
5. The resulting consequence is: A future actively consumed key-bearing VITE_ secret whose name evades the heuristic can be compiled into public JavaScript.
