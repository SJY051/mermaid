## Attack Path Steps

1. Attacker or external source controls the validated input: A local setup user invokes the repository bootstrap while the process umask and shared host determine resulting permissions.
2. The stated preconditions hold: The host is multi-user or otherwise exposes group/other reads, .env is newly created under a permissive umask, and credentials are filled in.
3. The closest control is reached but does not close the path: bin/setup.sh copies .env.example to .env without setting a restrictive mode.
4. The input reaches the validated sink: The newly created .env inherits ordinary copy/umask permissions and is later populated with service credentials.
5. The resulting consequence is: Other local users in the file's group or host may read API and database secrets when permissions resolve to 0644.

## Attack Path Facts
