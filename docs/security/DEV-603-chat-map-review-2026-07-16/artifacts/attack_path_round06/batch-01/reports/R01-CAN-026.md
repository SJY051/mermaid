## Attack Path Steps

1. Attacker or external source controls the validated input: An anonymous user can start a chat request and then activate New conversation.
2. The stated preconditions hold: A request is in flight, the user starts a new conversation, and then submits again before the abandoned server work ends.
3. The closest control is reached but does not close the path: The HTTP client supports AbortSignal, but chatSession omits it and newConversation only drops local state.
4. The input reaches the validated sink: The original server/provider call continues while the UI immediately permits a new request.
5. The resulting consequence is: One user can overlap abandoned expensive LLM and government calls, amplifying cost, quota use, and cold-path unavailability.
