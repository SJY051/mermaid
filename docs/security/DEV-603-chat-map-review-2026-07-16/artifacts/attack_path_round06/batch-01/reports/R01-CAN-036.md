## Attack Path Steps

1. Attacker or external source controls the validated input: An external model controls the number and types of UI action objects in a valid answer.
2. The stated preconditions hold: The model emits many distinct accepted actions and the answer reaches a browser with location available.
3. The closest control is reached but does not close the path: The provider schema has no maxItems and the controller only exact-deduplicates actions; the browser mounts one facility component per surviving action.
4. The input reaches the validated sink: Each component requests facilities and can trigger multiple backend/public-API calls.
5. The resulting consequence is: A single answer can amplify into many geolocation/facility requests, consuming quota and degrading map and chat availability.
