## Attack Path Steps

1. Attacker or external source controls the validated input: An anonymous client controls message objects labeled assistant in request history.
2. The stated preconditions hold: An anonymous caller submits an assistant-role history item and a user turn; the proxy accepts and forwards the request.
3. The closest control is reached but does not close the path: Emergency and allergy screening aggregate user-role text only, while ChatProxyService separately preserves client assistant-role history for the upstream model.
4. The input reaches the validated sink: Client-authored content is forwarded with role=assistant after the server's system context, rather than being normalized as untrusted user text.
5. The resulting consequence is: The proven impact is a role-boundary violation: client-controlled text is presented to the model as trusted prior assistant output and bypasses the user-only safety-screening input set. Harmful model obedience or unsafe final prose was not demonstrated.
