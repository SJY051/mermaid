# PR 가이드 — 브랜치에서 머지까지

이 문서는 **GitHub로 협업하는 게 처음인 팀원**을 위한 것입니다. PR(Pull Request)이 뭔지부터, 봇 리뷰를 읽고 고치는 법까지 — 우리 저장소의 실제 규칙대로 안내합니다. 막히면 [30분 규칙](#막히면)이 있습니다.

---

## 1. PR이 뭐고, 왜 쓰나 (1분)

`main` 브랜치는 **언제나 동작하는 코드**만 있는 곳입니다. 발표 날 시연하는 것도, 팀원이 아침에 클론하는 것도 전부 `main`이에요. 그래서 아무도 `main`에 직접 푸시하지 않습니다.

대신 **내 작업 공간(브랜치)**을 만들어 거기서 일하고, 끝나면 "이거 `main`에 합쳐도 될까요?"라고 요청을 올립니다. 그 요청이 **PR**입니다. PR이 올라가면:

1. **CI**(자동 검사)가 테스트 275 + 47개를 돌리고, 시크릿이 새지 않았는지 스캔합니다.
2. **Codex 봇**이 코드를 리뷰합니다 — 컨테이너에서 실제로 테스트를 돌려보면서요.
3. 사람이 확인하고 **머지**하면 `main`에 합쳐집니다.

즉 PR은 절차가 아니라 **안전망**입니다. 내 실수가 `main`에 닿기 전에 세 겹이 걸러줘요.

## 2. 두 가지 길

### 길 A — GitHub 웹에서 (문서·문구 수정: 터미널 불필요)

영어 문구, 문서, `synonyms.tsv` 검토처럼 파일 몇 개만 고칠 때는 브라우저로 충분합니다.

1. GitHub에서 파일을 열고 **연필 아이콘** 클릭
2. 고치기
3. 아래 커밋 화면에서 **"Create a new branch for this commit and start a pull request"** 선택 ← 이게 핵심입니다. "Commit directly to main"이 아니라요.
4. 브랜치 이름을 `docs/DEV-305-synonym-review`처럼 바꾸고 → **Propose changes** → **Create pull request**

### 길 B — 터미널에서 (코드 작업)

```bash
git switch main && git pull              # 최신 main에서 출발
git switch -c feat/DEV-207-detail-drawer # 브랜치 이름: <타입>/<작업ID>-<영어요약>

# ... 작업하고, 테스트 돌리고 ...

git add frontend/src/components/DetailDrawer.tsx   # 파일을 이름으로 지정. git add -A 금지
git commit -m "feat(facility): add the detail drawer skeleton"
git push -u origin feat/DEV-207-detail-drawer
```

푸시하면 GitHub 저장소 페이지에 노란 **"Compare & pull request"** 버튼이 뜹니다. 누르면 PR 작성 화면이에요.

### PR을 열 때

- **제목**은 커밋처럼: `feat(facility): add the detail drawer (UI-03)`
- **본문 템플릿**이 자동으로 뜹니다. 체크리스트는 **실제로 한 것만** 체크하세요. 안 하고 체크하는 건 거짓말이고, 어차피 CI와 봇이 잡습니다.
- 작업 ID(`DEV-207`)를 꼭 넣으세요. **추적성 자체가 채점 항목**입니다.
- 하나의 PR엔 하나의 일만. 리뷰어(사람이든 봇이든)가 한 번에 읽을 수 있는 크기로.

## 3. 봇 리뷰 — 돌리는 법

따로 돌릴 필요가 **없습니다.** PR을 열면 Codex가 자동으로 리뷰합니다. 그 외에 세 가지만 알면 됩니다:

| 하고 싶은 것 | 방법 |
|---|---|
| 리뷰 다시 요청 (커밋을 더 올린 뒤 등) | PR에 댓글: `@codex review` |
| 특정 관점으로 봐달라기 | `@codex review for security regressions` 처럼 뒤에 초점을 적기 |
| "이슈 없음"의 표시 | 봇이 코멘트 대신 **👍 반응**을 남깁니다 |

봇은 **P0와 P1만** 짚습니다. 사소한 스타일 지적은 설계상 하지 않아요. 그러니 **봇 코멘트가 떴다면 진짜 봐야 하는 것**입니다.

- **P0** = 머지하면 안 되는 문제. 안전 규칙([AGENTS.md §2](../AGENTS.md#2-invariants)) 위반이 여기 속합니다.
- **P1** = 머지 전에 고쳐야 하는 문제.

## 4. 봇 리뷰 — 읽고 고치는 법

봇 리뷰가 의미 있으려면 **사람이 읽고 고쳐야** 합니다. 이 순서를 권합니다:

**① 끝까지 읽기.** 코멘트에는 "어느 파일 몇째 줄이, 어떤 상황에서, 왜 문제인지"가 적혀 있습니다. 첫 문장만 보고 판단하지 마세요.

**② 확인하기.** 봇도 틀릴 수 있습니다. 코멘트가 말하는 파일을 열어 **주장이 코드와 맞는지** 직접 보세요. 참고로 지금까지 이 저장소에서 봇이 짚은 네 건(P1 둘, P2 둘)은 전부 진짜였습니다 — 하지만 그건 확인을 건너뛸 이유가 아니라, 확인할 가치가 있다는 증거입니다.

**③ 고치기 — 같은 브랜치에 새 커밋으로.** PR을 새로 열 필요 없습니다. 고치고 푸시하면 PR에 자동으로 쌓입니다. 그리고 `@codex review`로 다시 확인받으세요.

**④ 답 남기기.** 코멘트에 "고쳤습니다 → `abc1234`"라고 답하고, 봇 코멘트의 **👍/👎**도 눌러주세요(리뷰 품질 피드백이 됩니다).

**⑤ 동의하지 않으면 — 침묵시키지 말고 말하기.** 봇이 틀렸다고 생각되면 그 근거를 PR 댓글로 적고 팀에 물어보세요. **절대 하지 말 것:** 지적을 없애려고 테스트를 약화시키거나, 이해하지 못한 코드를 지우는 것. 그건 문제를 고치는 게 아니라 경보기를 끄는 겁니다.

> 봇이 뭘 기준으로 리뷰하는지 궁금하면 [AGENTS.md의 Review guidelines](../AGENTS.md#review-guidelines)를 보세요. **같은 기준을 PR 올리기 전에 내 diff에 먼저 적용**해보면, 봇이 짚기 전에 내가 먼저 잡습니다. 그게 제일 싼 리뷰예요.

## 5. 머지는 언제?

세 가지가 모이면 됩니다:

1. **CI 초록불** — 세 검사(backend / frontend / secret scan) 전부. `main`은 브랜치 보호가 걸려 있어 **이 조건 없이는 누구도(리드 포함) 머지할 수 없습니다.**
2. **봇 리뷰 해소** — 짚힌 것을 고쳤거나, 왜 안 고치는지 합의됐거나
3. **사람 승인** — 경로가 둘입니다:
   - **코드·설정·`synonyms.tsv`** → 리드(윤서진)가 승인·머지. tsv가 여기 속하는 이유: 문서처럼 보여도 **알레르기 차단 동작을 바꾸는 입력**이라서요 — 행 하나를 추가하면 새 차단이 생기고, 서명 칸은 사람이 검증했다는 기록입니다.
   - **문서·영어 문구** (docs/, README, 코멘트 문구) → **최정민(PM/QA)도 승인·머지할 수 있습니다.** 리드를 기다릴 필요 없어요.

**리드의 응답 약속:** 리드는 모든 PR에 **24시간 안에** 반응합니다 — 승인이든, 코멘트든, 최소한 "보고 있어요"든. 하루 넘게 조용하면 그건 리드가 규칙을 어긴 것이니, 편하게 찔러주세요.

머지되면 브랜치는 지워도 됩니다. 이미 `main`에 들어갔으니까요.
